Project Brief: Lighthouse Performance Tracking Dashboard

1. The Problem

Lighthouse (Chrome DevTools) only shows metrics for the one page you're currently testing. Every time you test a new URL, the old results are gone — there's no built-in history, no way to compare runs over time, and no way to track multiple websites at once.

2. The Goal (One-Line Prompt)


Build a web app where I enter a URL, it runs a Lighthouse audit, saves the results to a database, and shows me a dashboard with the history and trends for every website I've ever tested.



3. What the Project Actually Is

A mini performance-monitoring SaaS with three moving parts:

User enters URL
      ↓
Backend runs Lighthouse audit
      ↓
Extract & structure the metrics (JSON)
      ↓
Save results to database
      ↓
Frontend dashboard displays history, trends, comparisons

4. Recommended Tech Stack

LayerRecommendationWhyFrontendNext.js (React)Handles frontend + API routes in one appBackendNode.js / Next.js API routesRuns Lighthouse audits server-sideAudit enginelighthouse + chrome-launcher (npm)The actual audit runnerDatabaseNeon (serverless PostgreSQL)Structured data, generous free tier, scales wellORMPrisma or DrizzleType-safe DB access, easy migrationsChartsChart.js or RechartsTrend graphs, comparisonsScheduler (optional)Cron job / GitHub ActionsAuto re-scan sites on a schedule

5. Core Features (MVP)


Input a URL → trigger a Lighthouse audit
Store the results (scores + key metrics) in the database
List all previously tested websites
Click a website → see its full history as a timeline/graph
Compare the latest run to previous runs (spot regressions)


6. Stretch Features (v2+)


Scheduled/automatic daily audits (cron)
Alerts when a score drops below a threshold
Side-by-side comparison of two different websites
Multi-user accounts / teams (turns it into a real SaaS)
Public API to fetch a site's history externally
Queue system if you're scanning many URLs (Lighthouse is CPU-heavy, ~5–30s per run)


7. What Lighthouse Actually Gives You

Lighthouse returns a huge JSON report. You don't need to store all of it — here's what matters, grouped:

GroupContentsOverall scoresPerformance, Accessibility, Best Practices, SEO, PWA (0–100 each)Core Web VitalsFCP, LCP, CLS, TBT, Speed Index, TTIResource infoPage size, request count, JS/CSS/image/font sizesOpportunitiesSuggested fixes with estimated time savings (e.g. "reduce unused JS: 1.4s")DiagnosticsDOM size, main-thread work, JS execution time, critical request chainsAccessibility checksAlt text, contrast, ARIA, labels, heading orderSEO checksMeta description, canonical URL, structured data, mobile-friendlinessBest practices checksHTTPS, console errors, deprecated APIs, securityNetwork infoTransfer size, TTFB, request breakdownEnvironment infoLighthouse/Chrome version, timestamp, throttling settings

8. Suggested Database Schema

Use related tables rather than one giant table:

sql-- One row per monitored website
CREATE TABLE websites (
    id SERIAL PRIMARY KEY,
    url TEXT NOT NULL UNIQUE,
    title TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- One row per Lighthouse run
CREATE TABLE audits (
    id SERIAL PRIMARY KEY,
    website_id INTEGER REFERENCES websites(id),
    performance_score INTEGER,
    accessibility_score INTEGER,
    best_practices_score INTEGER,
    seo_score INTEGER,
    lighthouse_version TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Detailed numeric metrics per audit
CREATE TABLE metrics (
    id SERIAL PRIMARY KEY,
    audit_id INTEGER REFERENCES audits(id),
    fcp NUMERIC,
    lcp NUMERIC,
    cls NUMERIC,
    tbt NUMERIC,
    speed_index NUMERIC,
    tti NUMERIC
);

-- Optional: optimization suggestions per audit
CREATE TABLE opportunities (
    id SERIAL PRIMARY KEY,
    audit_id INTEGER REFERENCES audits(id),
    audit_name TEXT,
    estimated_savings_ms NUMERIC
);

This structure lets you query trends per website, compare runs, and extend the schema later without breaking anything.

9. Minimal Audit Runner (Node.js)

javascriptimport lighthouse from 'lighthouse';
import chromeLauncher from 'chrome-launcher';

async function runAudit(url) {
  const chrome = await chromeLauncher.launch({ chromeFlags: ['--headless'] });

  const options = {
    logLevel: 'info',
    output: 'json',
    port: chrome.port,
  };

  const result = await lighthouse(url, options);
  await chrome.kill();

  return {
    url,
    performance: result.lhr.categories.performance.score * 100,
    lcp: result.lhr.audits['largest-contentful-paint'].numericValue,
    fcp: result.lhr.audits['first-contentful-paint'].numericValue,
    cls: result.lhr.audits['cumulative-layout-shift'].numericValue,
    time: new Date(),
  };
}

This result object is what gets written into the audits + metrics tables above.

10. Realistic Constraints to Plan For


Lighthouse is CPU-heavy and takes ~5–30 seconds per run — don't run it synchronously on every page load; use a background job/queue if you scale up.
Running many audits concurrently will need a queue system (e.g., BullMQ + Redis) eventually.
